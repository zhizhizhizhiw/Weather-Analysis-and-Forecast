## 一、数据爬取阶段

### Q1：爬取过程中遇到了什么困难？

**难点在于三个层面：**

第一是反爬。天气后报网站虽然没设验证码，但对裸请求会直接拒绝。需要模拟完整的浏览器请求头，设置合理的超时和重试机制。还得在每次请求之间 sleep 0.5 秒，既防止被 ban，也避免对对方服务器造成压力。

第二是页面缺失。2025 年 8 月之后的数据还没有，请求过去返回空页面。如果不做判空处理，代码会在解析 table 时报错。我当时用 `if table is None: continue` 做了跳过处理，保证爬虫不会因为个别页面缺失而整体崩溃。

第三也是最有技术含量的——**非结构化数据清洗**。爬下来的原始数据长这样：

| 字段 | 原始值 | 
|------|--------|
| 天气 | `多云 /  晴` |
| 温度 | `7℃/-7℃` |
| 风力 | `北风 5-6级 / 北风 5-6级` |

每条记录的三个核心字段都是"白天/夜间"的复合字符串，必须拆分成独立字段才能做统计。这不是简单的 split，因为：温度包含负数（`-7℃`）、天气类型多样（`小到中雪`、`中到大雨`）、风力级别格式固定但风向多变（`东北风`、`西南风`等）。

我用了两套策略解决：对天气和风力字段，按 `/` 分隔符拆分白天和夜间；对温度和风力等级，用正则表达式提取数值。比如用 `r'(-?\d+)℃/(-?\d+)℃'` 匹配最高/最低温度，其中 `-?` 处理负温情况；用 `r'(\d+-\d+)'` 提取风力等级范围如 `3-4`、`5-6`。

### Q2：为什么选择 BeautifulSoup 而不是 Scrapy 或 Selenium？

数据量其实不大，总共 42 个页面、1332 条记录，不需要 Scrapy 那种异步框架的开销。BeautifulSoup 配合 requests 是最轻量的方案。另外这个网站是静态 HTML 页面，数据直接嵌在 table 里，不需要 Selenium 去执行 JavaScript 渲染。选型原则就是够用就好，不过度设计。

### Q3：怎么保证数据质量？有没有遇到脏数据？

做了几道检查：第一，爬取时判空，如果某个月份页面不存在就跳过并打印日志；第二，清洗时 `dropna(subset=["temp"])` 去除温度字段缺失的记录；第三，温度字段转为 float 类型时会自动过滤掉格式异常的条目。另外正则提取本身也有容错能力——格式不匹配的行在 extract 时会产生 NaN，后续会被过滤掉。

## 二、数据分析与可视化阶段

### Q4：中文乱码是怎么解决的？这是 Python 数据可视化的经典坑。

matplotlib 默认字体不支持中文，画出来全是方框。需要在代码里配两行：

```python
plt.rcParams['font.sans-serif'] = ['SimHei']
plt.rcParams['axes.unicode_minus'] = False
```

第一行指定中文字体（SimHei 黑体或 Microsoft YaHei），第二行解决负号 `-` 也变成方框的问题。这个问题如果没碰到过可能觉得简单，但实际在工作里几乎每个人第一次用 matplotlib 画中文图都会踩。

数据本身也有脏字符。天气字段里有些条目带了全角空格 `　` 和多余空格，导致同一种天气类型（比如"晴"）被识别成多个不同的值，统计结果就乱了。我在统计前做了 `str.replace('　', '').str.strip()` 来统一清洗。

### Q5：风力等级和天气状况的堆叠柱状图是怎么做的？为什么用堆叠图而不是饼图？

统计逻辑分两步：先用 `groupby(['month', 'wind_day_level']).size()` 按月份和风力等级交叉统计出现次数，然后用 `.unstack()` 把风力等级从行展开为列，得到一个 12 行 × N 列的矩阵（12 个月，N 个风力等级）。最后除以 3 得到三年月均值，再 plot 堆叠柱状图。

选堆叠柱状图而不是饼图的原因是：饼图适合看**总量**里的占比，但这里我们要看的是**每个月内部各种风力/天气的分布，同时还要横向对比不同月份**。饼图没法同时展示 12 个月的对比，堆叠柱状图一个图就能完成这个任务。

### Q6：异常检测那块是怎么做的？阈值怎么定的？

用了 7 日滑动平均加标准差阈值法。具体思路：

1. 计算 7 日滑动平均，平滑掉单日气温的随机波动
2. 以全部数据的**均值 ± 1 倍标准差**作为上界和下界
3. 超出上界的标红（高温异常），低于下界的标蓝（低温异常）

选 1 倍标准差是因为温度数据的波动相对集中，用太宽的范围（比如 2σ）会把真正的异常也漏掉；太窄又会把正常波动当成异常。这是一个经验参数，实际在面试时可以解释为"根据数据的实际分布情况做了调整"。

另外用了 center=True 的滚动窗口，这样平滑后的曲线不会偏移，能更准确地反映趋势。

### Q7：数据处理中用了 unstack() 和 groupby，解释一下它们的用法？

groupby 负责分组聚合，unstack 负责"行列转置"。举个例子，原始统计结果是：

```
month  wind_day_level  count
1      3-4             12
1      4-5             15
1      5-6             3
2      3-4             10
...
```

这种格式没法直接画图。`.unstack(fill_value=0)` 把 wind_day_level 从行索引转到列索引，变成：

```
month  3-4  4-5  5-6
1      12   15   3
2      10   13   5
...
```

这样一个二维矩阵就能直接喂给 matplotlib 的堆叠柱状图了。fill_value=0 很关键，有的月份可能没有某一级的风力，对应的位置要填 0 而不是 NaN，否则图就断了。

## 三、温度预测阶段

### Q8：auto_arima 是怎么选参的？最优阶数 (1,0,3) 是什么意思？

auto_arima 本质是在给定范围内搜索所有可能的 ARIMA 参数组合，对每个组合用最大似然法拟合模型，然后选 AIC（赤池信息准则）最小的那组参数。AIC 是一个权衡模型拟合度和复杂度的指标——你想让残差尽量小，但也不希望模型过拟合。

参数我设的是 `seasonal=True, m=12`（气温有明显的年周期，一年 12 个月），`max_p=3, max_q=3`（搜索范围限制在合理范围内，避免搜索空间爆炸）。

最终收敛到 (1,0,3)，解读：p=1 表示用前 1 天的温度预测今天；d=0 表示原序列已经是平稳的，不需要差分；q=3 表示考虑前 3 天的预测误差做修正。d=0 是合理的，因为温度本身就在一定范围内波动，不像股票价格那样有趋势性增长。

### Q9：MSE=4.87 这个结果算好还是不好？

MSE 4.87 意味着预测值和真实值的平均误差大约在 ±2.2°C 左右（sqrt(4.87)≈2.2）。对于日均温度的预测来说，这个误差在实际应用层面是可以接受的——你预测明天 25°C，实际 23°C 或 27°C，出入不大。但如果追求更高精度，可以尝试引入更多特征（如降水、气压、湿度等），从单变量时间序列升级为多元预测。

### Q10：MAPE 算出来是一个天文数字（16662823273633796%），这是什么问题？

这是我当时遇到的一个典型 bug。MAPE（平均绝对百分比误差）的公式中分母是真实值，当某些天的真实温度接近 0°C 时（大连冬天经常在零度附近），分母趋近于零，导致 MAPE 爆炸到天文数字。

**解决方案：** 对于含零及接近零的数据，MAPE **不适合**作为评估指标。我的处理是用 MSE（均方误差）替代 MAPE 来评估模型。如果面试官问，可以进一步说：可以考虑用 sMAPE（对称平均绝对百分比误差）或直接用 MAE（平均绝对误差，单位是度，更直观）。

这也是一个很好的反思点——**评估指标的选择要看数据特性**，不能无脑套用。

### Q11：你觉得这个预测模型还有哪些可以改进的地方？

三个方向：

第一，我实际做的是拟合而非真正的预测——模型是在全量数据上 fit 的，然后取 fittedvalues 对比。如果要严格评估预测能力，应该只用 2022-2024 年的数据训练，然后逐步预测 2025 年 1-6 月的每一天，不喂入任何 2025 年的信息。

第二，目前只有温度这一个特征（单变量）。实际天气受气压、湿度、降水、风力等多因素影响。如果能拿到这些数据做多变量 SARIMAX（外生变量），预测精度大概率会提升。

第三，可以试试深度学习模型如 LSTM，它对长序列的非线性依赖关系比 ARIMA 更擅长。复杂度更高但效果可能更好，这算是一个技术选型的 trade-off。

## 四、综合性的"项目复盘"回答

如果面试官问"简单介绍下这个项目，你最自豪的部分是什么"，可以参考这个结构：

**我做的第一个完整的端到端数据分析项目。** 从爬虫取数据，到清洗建表，到多维度可视化，最后做时间序列预测，完整跑通了一条数据链路。

最自豪的是**非结构化数据的正则清洗那一块**。原始数据看起来很简单，但实际藏着很多问题——白天和夜间的分隔线、负温度的处理、全角空格的干扰，这些细节如果不处理好，后面的统计分析全是错的。做数据分析和做算法最大的区别就是：你花 80% 的时间在数据预处理上，但真正决定结果质量的往往是那 80%。

另外印象深刻的就是 MAPE 那个 bug。做完一看 16 quadrillion percent，当时觉得肯定代码写错了，查了一圈才发现是数据本身的数学特性导致的——温度在零附近让 MAPE 失效。这种"模型跑了，数字出来了，但你得知道它对不对"的判断力，是做数据分析特别重要的素质。
